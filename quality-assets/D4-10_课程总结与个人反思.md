# D4-10 课程总结与个人反思

## 一、课程总览

《软件工程》课程通过四个递进式实验，模拟了真实软件项目的完整生命周期：

- **实验一**（需求工程与 UML 建模）：使用 Use Case、类图、时序图对 NekoCafé 平台进行需求分析，体会到"需求是软件的起点"——模糊的需求会导致后期的无尽返工。
- **实验二**（架构设计与微服务拆分）：将会员服务和预约服务拆分为独立微服务，实践了单一职责原则和 API 优先设计，理解了"高内聚、低耦合"在工程中的真正含义。
- **实验三**（DevOps 流水线与容器化部署）：搭建了从代码提交到生产部署的完整 CI/CD 流水线，涵盖 lint、test、build、scan、deploy 各阶段，并使用 Helm + Kubernetes 实现灰度发布。
- **实验四**（软件质量工程与 AI 辅助代码审查）：将前三个实验的产出置于质量显微镜下，通过测试金字塔、变异测试、AI 审查、缺陷分析等方法建立完整的质量保障闭环。

## 二、实验四核心收获

### 2.1 多维质量观

在接触 ISO/IEC 25010 之前，我对"软件质量"的理解局限于"功能是否正常"。通过本实验，我理解了质量的八大维度：
- **功能适用性**：不仅要做对，还要做全
- **可靠性**：异常场景下的优雅降级
- **性能效率**：P95 响应时间不仅仅是运维指标
- **安全性**：从 CORS 到 API 限流，安全是系统性问题
- **可维护性**：覆盖率不是终点，测试质量才是

### 2.2 测试金字塔的实践体会

"更多单元测试，更少 E2E 测试"这句原则在本实验中得到了验证。我为两个服务编写了 133+ 个测试用例，其中单元测试 64+69=133 个（含 PBT），集成测试 15 个，E2E 仅 3 条。金字塔形状天然形成——这并非刻意为之，而是遵循"哪里能最快定位问题就在哪里测试"的自然结果。

### 2.3 基于属性的测试 (PBT) 的惊艳

这是我最惊喜的收获。传统单元测试是"举例验证"——"当输入为 X 时，期望输出为 Y"。PBT 则是"不变量验证"——"对于所有合法输入，该属性恒成立"。以 `calculate_points` 为例，我不用枚举 1000 种输入组合，只需声明"输出始终在 [10, 5000] 区间内"，hypothesis 会自动搜索反例。这种思维的转变——从具体的例子到抽象的不变量——是我在本课程中最重要的认知升级。

### 2.4 变异测试验证测试质量

"覆盖率 ≠ 测试质量"这一句话，直到我亲身运行变异测试才真正理解。我的代码覆盖率达到了 94%，但第一个版本的变异分数只有 9.4%。这意味着大量测试虽然执行了代码行，却没有真正验证其逻辑正确性。通过针对性地添加边界值测试和等价类测试，变异分数提升到 66.7%。这让我明白：**写测试的目的不是覆盖代码，而是覆盖行为**。

### 2.5 AI 辅助代码审查的双面性

3 轮 AI 审查产出了 22 条意见，采纳率 86.4%。AI 在代码规范、安全最佳实践、弃用 API 检测方面表现出色，速度快且一致性好。但它也在架构决策（建议引入不必要的依赖注入框架）和业务上下文（不理解 MVP 优先级的权衡）方面暴露了局限。我的结论是：**AI 是最佳的第一轮审查者，但不是最终决策者**。AI + 人工审查的协作模式，是当前最优的质量策略。

## 三、遇到的挑战与解决

| 挑战 | 解决方案 |
|------|---------|
| hypothesis 测试中字符分类边界 | 改用 Unicode category 判断替代硬编码中文范围 |
| 变异测试工具 mutmut 路径解析问题 | 编写自定义变异测试脚本，使用 AST 变换 |
| httpx 版本与 starlette TestClient 不兼容 | 锁定 httpx 版本 < 0.28 |
| E2E 服务依赖启动顺序 | 添加健康检查等待 + 重试逻辑 |
| CI 中 flake8 --exit-zero 掩盖错误 | 移除该标志，lint 失败直接阻断 |

## 四、不足与改进方向

1. **PBT 覆盖深度不足**：虽达到了 5 类核心域类的 PBT 要求，但更多函数可以添加 PBT。例如 `can_cancel` 函数的时间边界可以用 hypothesis 的 `datetime` 策略更精确地测试。
2. **契约测试未使用真实 Pact Broker**：当前契约测试基于 pytest + TestClient，未集成 Pact Broker。在生产环境中，Pact Broker 可以提供契约版本管理和兼容性检查。
3. **E2E 测试未覆盖前端 UI**：当前 E2E 测试仅覆盖 API 层面，未使用 Playwright/Cypress 测试前端交互。
4. **安全测试深度不够**：OWASP ZAP 仅执行了 Baseline 扫描，未进行深度渗透测试。
5. **故障注入未实施**：Chaos Engineering 和 Postmortem 流程是加分项，但本次实验时间有限未完成。

## 五、对软件工程的再认识

经过四个实验，我对"软件工程"有了更深的理解：

- **需求是起点，但不是终点**——需求在开发过程中会被不断澄清和修正
- **架构是骨架，代码是血肉**——好的架构让测试和质量保障变得可能
- **测试是安全网，不是终点**——测试的价值在于提供"修改的勇气"
- **质量是文化，不是检查点**——CI/CD 门禁只是手段，真正的质量来自团队共识
- **AI 是工具，人是决策者**——AI 可以加速但不能替代工程判断

## 六、结语

本课程让我完成了从一个"会写代码的学生"到"有工程思维的开发者"的转变。我不再把"写完功能"当作终点，而是开始思考：这个功能有测试吗？边界情况考虑了吗？如果有人改了这段代码，测试能发现吗？生产环境出问题了我能在几分钟内定位？这些思维习惯，将伴随我整个职业生涯。

感谢课程组精心设计的四个递进式实验，让理论不再是空洞的概念，而是真实的工程实践。
